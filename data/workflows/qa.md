# QA — 测试 → 修复 → 验证循环

## 模式

| 模式 | 范围 | 适用场景 |
|------|------|----------|
| Quick | 仅 critical + high | 快速冒烟测试 |
| Standard | + medium | 默认 |
| Exhaustive | + low + cosmetic | 发版前 |
| Report-Only | 不修复，只报告 | 需要 bug 报告但不改代码 |

## Phase 1: 准备
- 检查工作区是否干净
- 如果在 feature branch：分析 git diff，识别受影响的页面
- 检测运行中的应用（常见端口）

## Phase 2: 测试执行
每个页面的检查清单：

### 视觉检查
- [ ] 布局在 375px / 768px / 1024px 下正常
- [ ] 无水平滚动
- [ ] 颜色对比度 ≥ 4.5:1
- [ ] 图片有 alt 文字

### 交互检查
- [ ] 所有按钮可点击（cursor: pointer）
- [ ] 表单提交有 loading 状态
- [ ] 错误提示在正确位置
- [ ] 空状态有引导

### 控制台
- [ ] 无 JS 错误
- [ ] 无 404 资源
- [ ] 无弃用警告

### 性能
- [ ] 首屏 < 3s
- [ ] 无大文件阻塞

## Phase 3: 健康评分

| 类别 | 权重 |
|------|------|
| 控制台错误 | 15% |
| 链接有效性 | 10% |
| 视觉一致性 | 15% |
| 功能正确性 | 20% |
| 用户体验 | 15% |
| 性能 | 10% |
| 内容准确性 | 10% |
| 无障碍 | 5% |

## Phase 4: 修复循环（Report-Only 模式跳过）

对每个可修复的问题：
1. 定位源文件
2. 最小修复（CSS 优先，不重构）
3. 原子提交：`fix(qa): ISSUE-NNN — 描述`
4. 前后对比截图验证
5. 写回归测试

**安全阀**：
- 每 5 个修复评估风险分（回退 +15%，跨 3+ 文件 +5%）
- 风险 > 20% → 停止修复，报告剩余
- 硬上限：50 个修复

## Phase 5: 最终报告
```
QA Report
├── 测试范围：{页面列表}
├── 健康评分：{before} → {after}
├── 已修复：{N} 个问题
├── 已推迟：{M} 个问题（附原因）
├── 回归测试：{通过/失败}
└── 发布就绪度：READY / READY_WITH_CONCERNS / NOT_READY
```
