# Agent Architecture Routing

当请求和 Agent 系统本身有关时，不要直接修改文件，先判断目标层。

核心目标：**先判断问题属于哪一层，再决定改哪个目录、哪些注册表、会影响哪些相邻层。**

## 何时使用

遇到以下请求时，优先先走本 skill：

- “这个能力该做成 command、skill 还是 agent 角色？”
- “这些角色边界有点乱，应该改哪一层？”
- “工具接入应该改插件、MCP 还是 skill？”
- “记忆该放项目内、知识库还是 wiki？”
- “路由老是选错角色，应该调哪里？”

## 决策流程

按顺序做这 5 步：

1. 判断变化对象  
   你要改变的是人格/角色、路由规则、执行流程、工具接入，还是知识落点？

2. 判断主要触发点  
   问题是出在入口定义、执行中间层、底层工具，还是沉淀层？

3. 判断影响范围  
   是全局生效、项目内生效，还是只影响某个角色/skill？

4. 判断连带修改  
   是否需要同步更新注册表、规范文件、说明文档？

5. 明确下一跳  
   给出：问题层级、目标目录、可能波及层、为什么不改其他层。

## 分层判断

### A. 角色层

关键词：
- 人格
- 角色
- SOUL
- command
- 身份

改动位置：
- `~/.claude/commands/`
- `~/.openclaw/agents/*/agent/`

适合：
- 角色定义
- 角色人格
- 角色职责说明
- 入口命令

### B. 路由 / 编排层

关键词：
- Hermes
- 路由
- 编排
- 技能边界
- 架构调整

改动位置：
- `hermes-loop`
- routing skills
- `SKILLS_REGISTRY.md`
- `~/.claude/CLAUDE.md`

适合：
- 任务分发
- 技能边界
- 角色协作规则
- 触发与优先级

### C. 执行技能层

关键词：
- skill
- workflow
- 触发词
- SOP

改动位置：
- `~/.claude/skills/`
- 项目内 `.claude/skills/`

适合：
- workflow
- 触发词
- SOP
- 输入输出格式

### D. 工具层

关键词：
- MCP
- 浏览器
- 命令执行
- 插件

改动位置：
- 工具配置
- 插件配置
- gateway / tool adapters

适合：
- MCP 接入
- 执行器能力
- 浏览器 / 插件 / adapter 配置

### E. 记忆 / 知识层

关键词：
- memory
- hindsight
- knowledge
- wiki

改动位置：
- Hindsight
- `knowledge-base`
- `knowledge-wikis`

适合：
- 记忆分层
- 长期知识落点
- wiki 沉淀策略

## 冲突处理

当一个问题看起来同时涉及多层时，按下面规则裁决：

1. 如果主要在“谁来做”，优先看角色层或路由层  
2. 如果主要在“怎么做”，优先看 skill 层  
3. 如果主要在“有没有这个能力”，优先看工具层  
4. 如果主要在“结果记哪”，优先看知识层  

不要因为最终会修改多个文件，就跳过分层判断。

## 输出要求

处理这类请求时，先明确：

1. 当前问题属于哪一层
2. 应修改哪个目录
3. 是否会影响其他层
4. 需要更新哪些注册表或规范文件

建议输出格式：

```md
## Agent Architecture Routing Decision
- 问题层级：
- 主改动位置：
- 可能波及层：
- 需要同步更新：
- 不改其他层的原因：
```

## 检查点

以下情况先确认用户：

- 你不确定改动应是全局还是项目内
- 你不确定这是角色问题还是 skill 边界问题
- 改动会影响多个角色或多个项目
- 会牵涉自动化入口、注册表、规范文件同步

## Fallback

如果判断不清，按更窄影响面处理：

- 先项目内，后全局
- 先 skill 层，后角色层
- 先路由说明，后底层工具

原则：**先在最小影响层验证，再向外扩散。**
