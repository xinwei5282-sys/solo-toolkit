# Knowledge Routing

这个 skill 只负责**判断信息应该落在哪里**，不直接承担完整写作、研究或实现。

核心目标：**把信息放到正确的层，而不是全塞进一个地方。**

## 何时使用

遇到以下请求时，先加载本 skill 再决定下一步：

- “这段内容应该记在哪里？”
- “这份总结要不要进知识库？”
- “这个结论是写进项目文档，还是沉淀成长期知识？”
- “这批资料应该做成 wiki、PRD、会议纪要，还是只保留当前会话？”
- “这个经验以后会不会重复用，值不值得做成 skill 或 SOP？”

## 五种落点

### 1. 会话层

适合：
- 临时上下文
- 最近对话线索
- 当前轮次判断
- 不值得复用的短期信息

不要放入长期层的典型例子：
- 一次性的澄清
- 当前回答里的临时假设
- 很快会失效的小结论

### 2. 项目层

适合：
- 当前项目需求
- 实现计划
- 开发进度
- 设计决策
- 当前迭代的行动项
- 只服务单个项目的调研结论

推荐去向：
- 项目内 `README` / `PRD` / `ARCHITECTURE`
- 任务拆解文档
- 设计记录 / 决策记录

### 3. 长期知识层

位置：
- `~/knowledge-base/`

适合：
- 原则
- 方法
- 踩坑
- 通用经验
- 可被多个项目复用的判断框架
- 稳定的最佳实践

### 4. 结构化研究层

位置：
- `~/knowledge-wikis/`

适合：
- 可持续扩展主题
- 需要交叉链接
- 会反复查询和编译
- 会不断追加来源、观点、案例的主题
- 需要形成“主题网络”的内容

推荐工具：
- `llm-wiki`

### 5. 技能 / 流程层

适合：
- 会被反复执行的固定工作流
- 稳定的 SOP
- 可抽象成 agent 能力的步骤化流程

推荐工具：
- `skill-creator`
- 更新现有 `SKILL.md`

## 决策流程

按顺序做这 5 步，不要跳步：

1. 先判断时效性  
   如果信息只在当前对话或当前几小时内有意义，优先留在会话层。

2. 再判断适用范围  
   如果只服务当前项目，进入项目层；如果能跨多个项目复用，继续往长期层判断。

3. 判断结构复杂度  
   如果内容会持续积累、需要多页面组织、需要交叉链接，进入结构化研究层；否则优先考虑长期知识层。

4. 判断复用方式  
   如果未来复用的不是“结论”，而是“流程本身”，优先沉淀为 skill / SOP，而不是普通笔记。

5. 明确下一跳  
   给出具体落点、推荐工具，以及为什么不放到其他层。

## 路由规则

### A. 只对当前回答有用

特征：
- 时效很短
- 价值主要体现在当前轮对话
- 之后大概率不会复查

去向：
- 会话层

### B. 只服务当前项目推进

特征：
- 和当前仓库、需求、设计、实现强绑定
- 离开当前项目后价值明显下降

去向：
- 项目层
- 必要时衔接 `prd-writer`、`summarize`

### C. 对多个项目有复用价值

特征：
- 是方法、原则、经验、踩坑
- 未来可能被再次检索

去向：
- `~/knowledge-base/`

### D. 会成长为持续研究主题

特征：
- 需要长期维护
- 资料来源多
- 需要章节化和交叉引用

去向：
- `~/knowledge-wikis/`
- `llm-wiki`

### E. 本质上是可复用流程

特征：
- 未来会重复执行
- 有稳定输入输出
- 可以写成步骤化说明

去向：
- 新建或更新 skill
- 必要时调用 `skill-creator`

## 冲突处理

当内容同时满足多个去向时，按下面优先级裁决：

1. 先看“主要消费者”是谁  
   当前项目团队优先放项目层；跨项目读者优先放长期层。

2. 再看“未来复用的是内容还是流程”  
   复用内容，进知识库；复用流程，进 skill。

3. 再看“是否需要结构化网络”  
   单篇可读即可，进知识库；需要持续扩展和交叉链接，进 wiki。

4. 仍然无法判断时  
   先落项目层或会话层，并明确标记“待升级”；不要为了长期沉淀而强行沉淀。

## 输出格式

完成路由后，至少输出这 4 项：

```md
## Knowledge Routing Decision
- 建议落点：
- 原因：
- 下一步动作：
- 不选其他层的原因：
```

如果用户明确要求你直接执行，也要先在心里完成这 4 项判断，再进入对应技能。

## 检查点

出现以下情况时，要先向用户确认再落长期层：

- 内容涉及隐私、账号、内部敏感信息
- 用户只是随口讨论，并没有要求沉淀
- 你不确定这条信息是否足够稳定
- 你不确定应该进项目层还是长期层

## Fallback

如果信息不完整，按下面方式处理：

- 无法判断时效性：默认先留在会话层
- 无法判断是否跨项目复用：默认先留项目层
- 无法判断是否值得结构化：先做单篇笔记，不直接建 wiki
- 无法判断是否该做 skill：先记为 SOP 草稿，等重复出现再升级

原则：**先放到不会造成信息污染的位置，再升级；不要一上来就沉淀到最高层。**

## 快速判断题

如果一条信息：

- 只对当前会话有用 → 不落长期
- 只对当前项目有用 → 进项目层
- 对多个项目有用 → 进 `knowledge-base`
- 会成长成一个主题网络 → 进 `knowledge-wikis`
- 会被重复执行为固定流程 → 考虑沉淀为 skill
