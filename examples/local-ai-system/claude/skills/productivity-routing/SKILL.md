# Productivity Routing

这个 skill 负责把“文档/汇报/整理/沉淀”类请求先分流。

核心目标：**先判断输出物是什么，再决定该走哪个技能、落到哪里、是否值得长期沉淀。**

## 适用请求

- 写文档
- 做汇报
- 整理信息
- 生成表格
- 形成 SOP
- 复盘总结
- 会议纪要
- 行动项梳理
- “帮我整理一下这些内容”

## 何时不该用

以下情况不要硬走效率整理流：

- 用户其实要做研究判断，不是整理材料
- 用户其实要改系统结构或 agent 架构
- 用户只是想要一个直接回答

这种情况下，优先考虑：
- `research-routing`
- `agent-architecture-routing`
- 普通会话回答

## 决策流程

按顺序做这 5 步：

1. 判断输出形态  
   是摘要、汇报、项目文档、表格、SOP、会议纪要，还是长期知识？

2. 判断服务对象  
   是当前项目团队、自己未来复用、跨项目读者，还是当前会话？

3. 判断复用价值  
   这是一次性整理，还是以后会反复复用的流程或模板？

4. 判断沉淀层级  
   结果应该留在项目层、长期知识层、结构化 wiki，还是 skill 层？

5. 明确下一跳  
   给出：主 skill、输出形态、落点位置，以及为什么不走其他路径。

## 路由规则

### A. 当前项目执行文档

去向：
- 项目目录
- `prd-writer`
- `summarize`

适合：
- 当前项目说明文档
- 需求汇总
- 任务拆解
- 里程碑汇报

### B. 通用经验总结

去向：
- `~/knowledge-base/`

适合：
- 通用方法论
- 踩坑总结
- 可跨项目复用的经验

### C. 结构化主题资料

去向：
- `llm-wiki`
- `~/knowledge-wikis/`

适合：
- 会持续扩展的主题资料
- 需要章节化和交叉链接的内容
- 未来会反复查询的专题

### D. 可重复流程

去向：
- 新建或更新 skill
- 更新 `SKILLS_REGISTRY.md`

适合：
- 稳定 SOP
- 固定输入输出的流程
- 重复出现的整理工作流

### E. 轻量摘要 / 纪要 / 要点提取

去向：
- `summarize`
- 必要时再交给 `knowledge-routing` 判断是否沉淀

适合：
- 会议纪要
- 长文摘要
- 行动项提取

### F. 表格 / 结构化清单输出

去向：
- 表格处理流
- 或直接生成结构化 markdown / csv

适合：
- 对比表
- 任务清单
- 里程碑追踪表

## 冲突处理

当一个请求既像“整理”又像“研究”时，按下面规则裁决：

1. 如果价值在“得出判断”，优先走研究路由  
2. 如果价值在“压缩和重组材料”，优先走效率整理流  
3. 如果结果会长期增长，优先升级到 `llm-wiki`  
4. 如果只服务当前项目，优先留在项目层，不强行建长期库

## 输出格式

路由后至少输出：

```md
## Productivity Routing Decision
- 输出形态：
- 主 skill：
- 建议落点：
- 下一步动作：
- 不走其他路径的原因：
```

## 关键判断

如果输出物主要是为了“以后重复用”，优先考虑沉淀 skill 或长期知识；  
如果主要是为了“当前项目推进”，优先写项目文档。

## 检查点

以下情况先确认用户：

- 你不确定这是一次性整理还是长期沉淀
- 你不确定结果该进项目层还是知识层
- 内容涉及敏感信息、内部会议、未公开材料

## Fallback

如果信息不完整，按更轻路径处理：

- 看不出是否长期复用：先按一次性整理
- 看不出是否值得建 wiki：先做单篇文档
- 看不出是否该做 skill：先记为 SOP 草稿

原则：**先把内容整理成可用结果，再决定是否升级为长期资产。**
