# Research Routing

这个 skill 只负责**分类和路由**，不直接承担完整执行。

核心目标：**先判断研究任务到底是什么，再决定该用哪个重技能、产出什么形态、落到哪里。**

## 用途

遇到以下请求时先加载本 skill：

- 查资料
- 做研究
- 竞品分析
- 行业分析
- 交易研究
- 知识整理
- 论文 / 文章梳理
- 市场 sizing
- 主题资料库建设
- “帮我先研究一下再说”

## 何时不该用

以下情况不要硬走研究工作流：

- 用户只是想要一个直接答案，不需要资料沉淀
- 问题是写作整理，不是研究判断
- 内容只服务当前任务，不值得结构化沉淀

这种情况下，优先考虑：
- 普通会话回答
- `summarize`
- `knowledge-routing`

## 决策流程

按顺序做这 5 步：

1. 判断研究目标  
   是为了市场判断、竞品比较、主题积累、项目支持，还是仅仅为了回答当前问题？

2. 判断输出形态  
   用户需要的是 brief、长期 wiki、项目结论、摘要，还是即时答复？

3. 判断时间跨度  
   这是一次性调研，还是未来会持续更新的研究主题？

4. 判断主要消费者  
   当前项目团队、长期知识库读者、投资/交易判断者，还是当前会话用户？

5. 明确下一跳  
   给出：主 skill、产出形态、落点位置、为什么不走其他路径。

## 路由规则

### A. 结构化长期主题研究

特征：
- 会持续更新
- 需要交叉链接
- 未来会反复查询

去向：
- `llm-wiki`
- `~/knowledge-wikis/`

典型场景：
- 行业长期观察
- 某个技术/主题的资料库
- 多来源、会不断追加的研究主题

### B. 项目内一次性调研

特征：
- 服务当前项目
- 结论主要在项目内使用

去向：
- 项目文档
- `summarize`
- 需要正式产品结论时，可衔接 `prd-writer`

典型场景：
- 当前功能立项前的背景调研
- 某个方案选择前的资料整理
- 给当前项目团队的结论性 memo

### C. 交易与市场研究

特征：
- 涉及股票、行业、标的、 thesis

去向：
- 即时判断 → 普通会话或当前专用分析流
- 长期 thesis / 持续跟踪 → 结构化 wiki

说明：
- 如果只是当下判断，不必强行建长期研究库
- 如果会持续跟踪同一标的/行业，再升级为长期主题研究

### D. 市场 / 用户 / 赛道研究

特征：
- 关心市场规模、用户分层、需求验证、行业趋势
- 目标是支撑产品或商业判断

去向：
- `market-research`
- 结果按需要落项目层或长期研究层

### E. 竞品 / 定位 / 差异化研究

特征：
- 关心 competitor landscape、feature comparison、pricing、positioning
- 目标是判断如何竞争或差异化

去向：
- `competitive-brief`
- 如果要长期维护竞品库，再配合 `llm-wiki`

### F. 普通问答型查询

特征：
- 只需要当前回答
- 不值得沉淀结构化知识

去向：
- 普通会话回答

### G. 论文 / 长文 / 资料摘要

特征：
- 用户主要想压缩内容
- 需要的是提炼、总结、提取结论
- 不一定需要完整研究框架

去向：
- `summarize`
- 如后续形成长期主题，再升级为 `llm-wiki`

## 冲突处理

当一个请求同时像“研究”和“整理”时，按下面规则裁决：

1. 如果价值在“判断”，优先走研究 skill  
   如市场判断、竞品结论、主题结论。

2. 如果价值在“压缩材料”，优先走 `summarize`  
   如读论文、读长文、提取要点。

3. 如果价值在“长期积累”，优先走 `llm-wiki`  
   如会持续更新、交叉链接、多次查询。

4. 如果只服务当前项目，优先落项目层  
   不要为了显得正式而强行建长期 wiki。

## 输出格式

路由后至少明确这 5 项：

```md
## Research Routing Decision
- 研究类型：
- 主 skill：
- 输出形态：
- 建议落点：
- 不走其他路径的原因：
```

## 产出要求

路由后至少明确三件事：

1. 主角色是谁
2. 主 skill / 主知识层是谁
3. 最终结果应落在哪个存储层

如果你已经能明确这三件事，下一步就直接切到对应 skill 执行，不要在路由阶段停太久。

## 检查点

出现以下情况时，要先向用户确认：

- 你不确定这是一次性结论还是长期研究主题
- 你不确定用户要“研究判断”还是“资料摘要”
- 内容涉及敏感资料、闭源文档、内部材料
- 你不确定结果该落项目层还是长期知识层

## Fallback

如果信息不完整，默认按更轻的路径处理：

- 看不出要长期维护：先按一次性调研
- 看不出要结构化积累：先按摘要或普通回答
- 看不出是市场还是竞品：先问“你更关心市场判断还是竞争对手比较？”
- 看不出是否需要沉淀：先回答当前问题，不强行建库

原则：**先用最轻、最不污染知识库的路径完成任务；有持续价值再升级。**
