# Investigate — 系统化根因调试

**铁律：没有根因分析，不做修复。**

## Phase 1: 收集症状
- 错误信息、堆栈跟踪、复现步骤
- 读代码追踪路径
- `git log` 检查近期变更
- 确定性复现

## Phase 2: 模式匹配
常见 bug 模式：
- 竞态条件
- nil/undefined 传播
- 状态污染
- 集成失败（API 变更、schema drift）
- 配置漂移
- 缓存过期

## Phase 3: 假设验证
1. 在疑似根因处加 log/断言
2. 运行复现 — 证据是否匹配？
3. **三振出局**：连续 3 个假设失败 → 升级给用户

## Phase 4: 修复
- 修根因，不修症状
- 最小 diff（最少文件、最少行数）
- 写回归测试：**没有 fix 时必须失败，有 fix 时必须通过**
- 运行全套测试 — 不允许回归
- 修改超过 5 个文件 → 先问用户

## Phase 5: 验证报告
```
DEBUG REPORT
├── 症状：{原始错误描述}
├── 根因：{具体原因}
├── 修复：{做了什么}
├── 证据：{测试输出/日志}
├── 回归测试：{测试名}
├── 相关项：{TODOS/已知问题}
└── 状态：DONE / DONE_WITH_CONCERNS / BLOCKED
```
